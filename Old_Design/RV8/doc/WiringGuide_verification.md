# WiringGuide Verification Report

**Date:** 2026-05-15  
**Verified against:** `rv8_cpu.v` (Verilog behavioral model)  
**Verdict:** ❌ NOT BUILDABLE AS-IS — Multiple critical issues

---

## 1. Pin Assignment Completeness

### ❌ FAIL — Many chips have unassigned pins

| Chip | Missing Pins | Severity |
|------|-------------|----------|
| U4 (74HC161) | Pin 15 (TC) floating — no carry out | Low (PC won't exceed 16 bits anyway) |
| U8 (74HC574, t0) | Pins 12-19 (Q0-Q7) ALL BLANK | **CRITICAL** — t0 outputs go nowhere |
| U9 (74HC574, sp) | Pins 12-19 (Q0-Q7) ALL BLANK | **CRITICAL** — sp outputs go nowhere |
| U10 (74HC574, pg) | Pins 12-19 (Q0-Q7) ALL BLANK | **CRITICAL** — pg outputs go nowhere |
| U13 (74HC283) | Pins 1,2,4,10,13 listed as blank in netlist | Medium — comments say connected but pin map doesn't match |
| U14 (74HC283) | Pins 1,2,4,6,10,11,13,15 listed as blank | **CRITICAL** — high nibble B inputs not wired |
| U18 (74HC138) | Pins 7,9-15 (all /Y outputs) unconnected | Medium — decode outputs unused |
| U22 (74HC08) | ALL gate pins blank | Medium — chip allocated but unused |

### What's missing:
- **U8 Q outputs** need to connect to: ALU B-input mux (when ALU source = t0), data bus (for PUSH t0), address mux (for t0 as address source)
- **U9 Q outputs** need to connect to: address mux high-byte (stack page $30:sp), ALU A-input (for sp arithmetic)
- **U10 Q outputs** need to connect to: address mux high-byte (for pg:imm addressing)

### Fix:
- Route U9.Q[7:0] to address mux B-inputs (needs a SECOND address mux layer for high byte — see Issue #6)
- Route U10.Q[7:0] to high-byte mux
- Route U8.Q[7:0] to ALU B-input via additional mux or tri-state buffer
- **Adds chips:** Yes, at minimum 1× 74HC245 (tri-state for t0→bus) + high-byte mux (see #6)

## 2. Bus Contention Analysis

### ⚠️ POTENTIAL CONTENTION

| Issue | Details |
|-------|---------|
| ALU result vs Data bus | ALU outputs (U13/U14 sum pins) connect DIRECTLY to U7 (a0) data inputs. They do NOT connect to the data bus. This means **no contention** but also **no way to write ALU results to memory** (see Issue #7). |
| U5/U6 IR outputs | /OE=GND (always driving). U5 drives control Flash address lines (OK, dedicated). U6 drives ALU B inputs via XOR (OK, dedicated path). No contention. |
| U7 (a0) outputs | /OE=GND (always driving to ALU A inputs). Dedicated path, no contention. |
| U19 (bus buffer) | DIR and /OE controlled by control signals. **BUT:** During writes, CPU must drive data bus. Who drives it? U7 (a0) has /OE=GND but outputs go to ALU, not data bus. **There is no path from any register to the external data bus for store operations.** |

### ❌ CRITICAL: No write-data source on external bus
The Verilog model has `data_out` mux selecting between `reg_rd_data`, `pc_out[15:8]`, `pc_out[7:0]`, flags, and `a0_out`. The wiring guide has NO equivalent — no tri-state buffer or mux connecting register outputs to the data bus B-side of U19.

### Fix:
- Add 74HC245 (U25) as register-to-bus buffer, controlled by `mem_wr` signal
- OR: Use U8/U9/U10's /OE pins (currently GND) — make them active only when that register is selected for output, and route Q outputs to data bus
- **Adds chips:** 1× 74HC245 minimum, plus a 74HC138 or 74HC139 to select which register drives the bus

## 3. Control Signal Traceability

### ❌ FAIL — Multiple signals have no source or destination

| Signal | Status | Problem |
|--------|--------|---------|
| `a0_clk` | ❌ No source | Listed on U7.11 but not generated by control Flash or any gate |
| `t0_clk` | ❌ No source | Listed on U8.11 but not generated anywhere |
| `sp_clk` | ❌ No source | Listed on U9.11 but not generated anywhere |
| `pg_clk` | ❌ No source | Listed on U10.11 but not generated anywhere |
| `pl_ld` | ❌ No source | Listed on U11.9 but not generated anywhere |
| `ph_ld` | ❌ No source | Listed on U12.9 but not generated anywhere |
| `ie_clk` | ❌ No source | Listed on U21.11 but not generated anywhere |
| `ie_data` | ❌ No source | Listed on U21.12 but not generated anywhere |
| `sub_mode` | ⚠️ Vague | "derived from alu_op" — no actual gate/connection specified |
| `carry_in` | ⚠️ Vague | "derived from alu_op + flag_c" — no actual gate/connection specified |
| `flags_we` | ⚠️ Vague | "from control" — not assigned to any specific Flash output bit |
| `sp_inc/sp_dec` | ⚠️ Vague | "from control" — not assigned to any specific Flash output bit |
| `ptr_inc` | ⚠️ Partial | Listed as "from control" but also shows U11.7/10 + U12.7 — OK for destination but no source pin on Flash |
| `bus_dir/bus_oe` | ⚠️ Vague | "from control" — not assigned to specific Flash output |
| `alu_op[3:0]` | ⚠️ Split | D7 + "ctrl2.D0-D2" — ctrl2 is mentioned but never defined as a chip |
| `addr_sel1` | ⚠️ No destination | ctrl2.D4 but only addr_sel0 connects to U16.1/U17.1 |
| `wr_sel[2:0]` | ⚠️ No destination | ctrl2.D5-D7 but no decode logic to generate individual register clocks |

### The "ctrl2" problem:
The wiring guide mentions `ctrl2.D0` through `ctrl2.D7` for 8 more control signals, implying a **second control Flash chip**. But:
- No second SST39SF010A is listed in the parts
- No pin assignments for a second Flash
- The chip count says "23 CPU + Control Flash" (singular)

### Fix:
- **Option A:** Add a second SST39SF010A (CTRL2) with identical address inputs, providing 8 more output bits. This gives 16 control bits total — enough for all signals.
- **Option B:** Use the single Flash with a 2-phase scheme (latch D[7:0] on CLK phase 1, read second byte on phase 2). Requires a 74HC574 latch + phase logic.
- **Option C:** Derive `a0_clk`, `t0_clk`, `sp_clk`, `pg_clk` from `reg_we` + `wr_sel[2:0]` through a 74HC138 decoder.
- **Adds chips:** Option A: +1 SST39SF010A. Option C: +1 74HC138 + 1 74HC574 (for latching wr_sel from Flash)

## 4. Control Flash Address Bits

### ✅ PASS (with caveat)

The CTRL pin map shows:
- A0-A7 = opcode[7:0] (8 bits)
- A8-A12 = state[4:0] (5 bits)  
- A13 = flag_z
- A14 = flag_c
- A15 = flag_n
- A16 = GND

**Total used: 16 address bits** (A16 tied to GND = 1 bank of 64K entries)

The SST39SF010A has 17 address pins (A0-A16), 128KB capacity. Using 16 bits = 64KB. ✅ Sufficient.

### Caveat:
The Bus section header says `{state[4:0], opcode[7:0], flags[2:0]} = 16 bits` which matches. However, the Verilog `rv8_control` only uses a 3-bit state (states 0-6). The wiring guide allocates 5 bits for state (32 states). This is **over-provisioned** but not wrong — it wastes Flash space but works.

## 5. Control Flash Data Outputs

### ❌ FAIL — 8 outputs insufficient

The single Flash provides 8 data bits (D0-D7). The wiring guide assigns:
- D0 = /RD
- D1 = /WR  
- D2 = pc_inc
- D3 = pc_ld
- D4 = ir0_clk
- D5 = ir1_clk
- D6 = reg_we
- D7 = addr_sel (1 bit only)

**Signals that NEED to come from control but have no Flash output:**
1. `alu_op[3:0]` — 4 bits (only 1 bit fits in D7 area)
2. `addr_sel1` — 1 bit (only addr_sel0 assigned)
3. `wr_sel[2:0]` — 3 bits
4. `flags_we` — 1 bit
5. `sp_inc` — 1 bit
6. `sp_dec` — 1 bit
7. `ptr_inc` — 1 bit
8. `bus_dir` — 1 bit
9. `bus_oe` — 1 bit
10. `sub_mode` — 1 bit (could derive from alu_op)
11. `carry_in` — 1 bit (could derive from alu_op + flag_c)

**Minimum additional bits needed: 13** (after deriving sub_mode and carry_in from alu_op)

With 8 bits from one Flash, you're **13 bits short**. Even with a second Flash (8 more bits = 16 total), you'd need to pack carefully or add a third output source.

### Fix:
- **Add second SST39SF010A** (same address bus, 8 more data outputs) → 16 bits total
- Derive `sub_mode` from `alu_op[0]` via gate (saves 1 bit)
- Derive `carry_in` from `alu_op[0] & flag_c` via AND gate (saves 1 bit)  
- Derive `bus_dir` from `/WR` inverted (saves 1 bit)
- Derive `bus_oe` from `/RD | /WR` (active during any memory access, saves 1 bit)
- **Packed into 16 bits:** D0=/RD, D1=/WR, D2=pc_inc, D3=pc_ld, D4=ir0_clk, D5=ir1_clk, D6=reg_we, D7=alu_op0, D8=alu_op1, D9=alu_op2, D10=alu_op3, D11=addr_sel0, D12=addr_sel1, D13=wr_sel0, D14=wr_sel1, D15=wr_sel2 — then derive flags_we, sp_inc, sp_dec, ptr_inc from alu_op+state combinations
- **Adds chips:** +1 SST39SF010A (CTRL2), +1 74HC08 or 74HC32 for derived signals

## 6. Address High Byte (A[15:8]) Routing

### ❌ CRITICAL FAIL — No high-byte mux exists

The Verilog model has an address mux with 8 sources:
```
3'd0: PC (full 16-bit)
3'd1: ptr_out (ph:pl)
3'd2: {$30, sp}     — stack
3'd3: {$00, imm}    — zero-page
3'd4: {pg, imm}     — page-relative
3'd5: {$FF, imm}    — vector
3'd6: {$30, sp+imm} — sp-relative
3'd7: {$30, sp-1}   — stack pre-dec
```

The wiring guide only has:
- **U16/U17 (74HC157×2):** 2-input mux selecting between PC[7:0] and pointer pl[7:0] for A[0:7]
- **U3/U4 outputs:** PC[15:8] goes DIRECTLY to ROM address pins (hardwired!)
- **NO mux for A[8:15]** — the high byte is always PC[15:8] for ROM, and... nothing for RAM

### What's broken:
1. **Stack access ($30xx):** No way to put $30 on A[15:8]. The sp register outputs aren't even connected (Issue #1).
2. **Zero-page ($00xx):** No way to force $00 on A[15:8].
3. **Page-relative (pg:imm):** pg register outputs aren't connected (Issue #1), and no mux to route them.
4. **Pointer high (ph):** U12 outputs go to U17 B-inputs, which only covers A[4:7]. The pointer high byte should provide A[8:15] but there's no path.
5. **Vector fetch ($FFxx):** No way to put $FF on A[15:8].

### The low-byte mux is also incomplete:
- U16/U17 are 2-input muxes (74HC157, select A or B). The Verilog needs 8 address sources for the low byte too (sp, imm, sp+imm, sp-1, pl). Only PC and pl are wired.

### Fix:
**This requires a complete address mux redesign:**

For A[7:0] (low byte):
- Replace U16/U17 (74HC157, 2:1) with 74HC251 (8:1 mux) ×8, OR
- Use 74HC153 (4:1 mux) ×4 for low nibble + ×4 for high nibble = 8 chips, OR
- Use 2× 74HC245 tri-state buffers (one for PC low, one for pointer low, one for sp, one for operand) with active-low enables from a 74HC138 decoder

For A[15:8] (high byte):
- Need a mux/tri-state scheme to select between: PC[15:8], ph, $30, $00, pg, $FF
- Minimum: 74HC245 for PC high + 74HC574 latch loaded with page constant from control + tri-state enable logic
- OR: 74HC153 ×4 (4:1 mux for 4 sources) with addr_sel[1:0] selecting {PC_high, ph, pg, fixed_page}
- The "fixed page" ($30, $00, $FF) could come from the operand/control Flash directly

**Adds chips:** +4 to +8 chips depending on approach. This is a **major architectural gap**.

## 7. ALU Result → Data Bus Path

### ❌ FAIL — No path exists

The ALU (U13/U14) sum outputs connect ONLY to U7 (a0) data inputs:
- U13.S1→U7.2, U13.S2→U7.3, U13.S3→U7.4, U13.S4→U7.5
- U14.S1→U7.6, U14.S2→U7.7, U14.S3→U7.8, U14.S4→U7.9

There is **no connection** from ALU result to:
- The data bus (for memory writes of computed values)
- Other registers (t0, sp, pg, pl, ph)

In the Verilog model, `reg_wr_data` feeds ALL registers, and `data_out` can put `a0_out` on the bus. But in hardware:
- a0 outputs (U7.12-19) go to ALU A-inputs, not to the data bus
- No tri-state buffer connects a0 to the data bus

### What's needed:
1. **a0 → data bus:** For STA (store accumulator) operations
2. **ALU result → other registers:** For LI t0, sp arithmetic, etc.
3. **ALU result → data bus:** For computed store operations

### Fix:
- Add 74HC245 (U25): A-side = a0 outputs (U7.12-19), B-side = data bus, DIR=LOW (A→B), /OE = store_enable
- For ALU→other registers: Route ALU outputs to a shared "result bus" that connects to all register D-inputs via the data bus (requires the registers to load from data bus, which U8/U9/U10 already do — pins 2-9 = D0-D7)
- The trick: during register-write cycles, put ALU result on data bus via U25, then clock the target register
- **Adds chips:** +1 74HC245

## 8. Zero-Detect Circuit

### ❌ FAIL — No zero-detect hardware exists

The wiring guide says:
```
alu_zero : NOR of all ALU result bits → U20.2 (D1)
```

But **no NOR gate or zero-detect circuit is actually wired**. There is no chip assigned to perform the 8-input NOR function.

An 8-input NOR requires:
- The ALU result bits (U13.S1-S4 + U14.S1-S4) as inputs
- A circuit that outputs HIGH only when ALL 8 bits are LOW

### Fix options:

**Option A: 8-input NOR from discrete gates (uses existing + new chips)**
- Use 74HC02 (quad NOR): NOR pairs of bits, then NOR the results, then AND
- Actually need: 74HC30 (8-input NAND) + inverter, OR 74HC4078 (8-input OR/NOR)
- 74HC4078: single chip, 8-input OR with complementary outputs. Pin 13 = NOR output = zero flag
- **Adds:** +1 74HC4078 (or 74HC30 + 1 gate from existing spare)

**Option B: Use 74HC688 (8-bit comparator)**
- Compare ALU result with $00. P=Q output = zero flag.
- Overkill but clean. 
- **Adds:** +1 74HC688

**Option C: Cascaded OR + invert using existing gates**
- U22 (74HC08) is completely unused — but AND gates can't make OR/NOR
- U23 (74HC32) is fully used for address decode
- No spare NOR or OR gates available
- **Cannot use existing chips**

**Recommended:** +1 74HC4078 (8-input NOR gate, single 14-pin DIP)
- Inputs: U13 pins 4,1,13,10 + U14 pins 4,1,13,10
- Output: → U20.2 (flag_z D input)

## Conclusion

The WiringGuide represents an **early draft** that correctly identifies the chips needed for the core datapath (PC, IR, ALU adder, pointer) but is **incomplete for a functioning CPU**. The critical gaps are:

1. **No high-byte address mux** — makes stack, zero-page, and page-relative addressing impossible
2. **No register-to-bus path** — makes store operations impossible  
3. **Only 8 control outputs** — needs 16+ for all control signals
4. **No zero-detect** — Z flag never updates
5. **Registers t0/sp/pg have no output routing** — they can be loaded but never read

The design needs a **revision pass** focusing on:
1. Complete address mux (both bytes, all modes)
2. Data bus write path (register → bus)
3. Second control Flash + register clock decode
4. Zero-detect circuit
5. High-nibble XOR for SUB (only low nibble has U15)
